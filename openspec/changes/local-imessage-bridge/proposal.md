# 提案：local-imessage-bridge

## 问题定义

项目需要一套完全运行在本机的方式，把 iMessage 作为 Codex 的长期对话入口。bridge 必须能够接收白名单联系人发送的消息，维护多轮会话上下文，支持带图片输入的对话，把内容交给本地 Codex 处理，并通过 Messages.app 把结果发回去，同时避免引入不必要的云端编排复杂度。

## 目标

- 让一对一 iMessage 对话能够驱动本地的持续 Codex 工作流
- 在多条消息之间保留多轮会话，而不是把每条请求当作一次性执行
- 支持多个白名单联系人，而不是只支持一个固定联系人
- 当 iMessage 含有图片附件时，允许图片作为输入进入 Codex
- 保持系统完全本地、结构清晰、便于在单机上调试
- 在尽量低摩擦使用的同时，保留对回环和未授权联系人的保护

## 第一版非目标

- 通用消息平台集成
- 多用户认证、租户体系、远程设备同步
- 第一版自动参与群聊
- 第一版支持多个 agent provider
- 第一版支持完整的富媒体出站能力
- 第一版支持通过 iMessage 动态修改白名单和 workspace 映射

## 约束

- bridge 运行在具有 Messages 数据访问权限和自动化权限的 macOS 机器上
- iMessage 集成依赖 `imsg` 作为本地传输层
- 为了获得良好的使用体验，必须支持持久化 Codex 会话
- 图片支持取决于 Codex 集成面支持的输入协议，以及 bridge 如何在本地暂存附件

## 已确认决策

- 系统只接入 `Codex`，不在第一版支持其他 provider
- 系统使用 `imsg` 作为 iMessage 传输层
- 系统优先对接 `Codex app-server` 作为持久线程承载面，`codex exec` 仅作为降级恢复路径
- 系统偏向低摩擦执行，不增加显式确认流
- 系统必须具备回环保护和发送者过滤
- 系统支持可配置的 iMessage 白名单联系人
- 非白名单联系人收到固定拒绝回复，引导其联系管理员
- 每个白名单联系人都映射到一个独立的持久 Codex 线程
- 第一版图片能力只支持“图片输入理解”，不支持图片生成后回传
- 每个白名单联系人都映射到一个默认本地 workspace 目录
- 通过 iMessage 动态管理白名单和 workspace 映射属于第二版能力
- 系统支持在短时间窗口内合并同一联系人的连续消息后再提交给 Codex
